　　摘要：随着信息化进程的加速，跨部门、跨组织的数据跨域交换已成为支撑
业务协同的核心环节 。然而，传统的点对点交换方式由于耦合度高、扩展性差，
难以应对复杂的跨域安全隔离与大规模异构环境 。针对上述问题，本文提出并
设计了一种基于发布-订阅（Publish/Subscribe）机制的数据跨域交换方法 。
　　本研究利用发布-订阅模式天然的异步解耦与多对多通信特性，构建了一个
安全、可扩展的跨域数据交换平台原型 。在架构设计上，系统采用 MQTT 协
议作为核心通信引擎，利用其轻量级主题订阅机制与服务质量（QoS）保障，降
低了数据生产者与消费者的耦合度 。针对跨域场景中的安全风险与网络不稳定
问题，本文通过 ACL（访问控制列表）权限控制、TLS 端到端加密、QUIC以
及基于消息持久化的重试策略，确保了数据在异构域间传输的安全性和可靠性 。
　　在系统实现方面，本文采用 Spring Boot 3 框架构建后端管理模块，结合 
Docker 容器技术模拟网络隔离环境，并通过 Linux Traffic Control 工具模拟弱网
环境以验证系统的鲁棒性 。前端利用 React 与 ECharts 实现了数据流向的可视
化监控与拓扑展示 。实验结果表明，该方法能够有效打破"数据孤岛"，在保证
域间隔离的前提下，实现高效、灵活的数据流通 。本研究不仅为跨域数据共享
提供了一种可行的技术路径，也为政务协同、供应链管理等实际应用场景提供了
重要的参考价值 。

Data Cross-Domain Exchange Method Based on 
Publish/Subscribe Mechanism

　　ZHANG Yunyi 
School of Computer and Information Science, School of Software, Southwest 
University, Chongqing 400715, PR China
　　Abstract: As informatization accelerates, cross-domain data exchange has 
become essential for business collaboration. Traditional point-to-point methods, due 
to high coupling and poor scalability, struggle with complex cross-domain security 
and heterogeneous environments. This paper proposes a data cross-domain exchange 
method based on a Publish/Subscribe mechanism.
　　Using the inherent decoupling of the pub/sub pattern, we design a secure and 
scalable platform prototype. The architecture employs MQTT as the core 
communication engine, reducing producer-consumer coupling through its lightweight 
topics and QoS. For cross-domain security and network reliability, we implement 
ACL control, TLS encryption, and message persistence with retry mechanisms.
　　The system is built with Spring Boot 3 for the backend, uses Docker to simulate 
isolated networks, and applies Linux Traffic Control to test robustness under weak 
networks. The frontend, developed with React and ECharts, enables visual flow 
monitoring. Results show the method effectively breaks data silos, enabling secure 
and efficient cross-domain data flow. This work provides a practical technical 
approach for scenarios like government collaboration and supply chain management.

Keywords: Publish/Subscribe; Cross-Domain Data Exchange; MQTT; Access 
Control; Software Engineering

 
第一章 绪论
1.1 研究背景
1.1.1 信息化社会中数据孤岛问题日益突出
　　随着信息化社会的快速发展，数据已成为核心生产要素和战略资源。根据《中
共中央 国务院关于构建更加完善的要素市场化配置体制机制的意见》（2020）
明确要求，数据要素需 "破除行政性垄断和部门壁垒，推动跨域共享共用"[6]，
但现实中全球数据总量指数级增长的同时，大量数据却被封闭在不同的组织、部
门或系统中，形成了一个个"数据孤岛"。这些孤岛导致数据无法有效流通和共享，
严重制约了数据的潜在价值发挥。例如，政府各部门间的数据分割使得政务服务
难以实现"一网通办"；企业供应链上下游的信息不对称影响了协同效率；医疗机
构间的患者信息孤立则阻碍了诊疗水平的整体提升。数据孤岛问题的根源在于系
统异构、部门壁垒以及安全隐私顾虑，导致数据虽多却难以互联互通，成为了制
约数字经济和智慧社会建设的关键瓶颈。正如 Chen（2025）在 Privacy Passport 
研究中指出的 [1]，跨域数据共享的本质矛盾在于协作需求与隐私泄露风险之间
的博弈，这要求系统必须具备严密的隐私保护机制。

1.1.2 跨域数据交换需求的快速增长
　　在数字化转型浪潮下，跨域数据交换需求呈现爆发式增长。在政务协同领域，
不同行政部门或地域间需要共享人口、税收、信用等数据，以支撑"放管服"改革
和跨省通办服务；在供应链管理领域，上游供应商、下游分销商以及物流企业间
需实时交换订单、库存和物流信息，以实现精准预测和高效协同；在医疗信息共
享领域，各级医疗机构间患者病历、影像和检验数据的跨域流动，可显著提升诊
断准确性和公共卫生应急能力。同时，在智慧城市、工业互联网等新兴领域，跨
域的数据交互需求同样迫切。根据《"十四五" 数字经济发展规划》明确要求，
需推进数据跨部门、跨层级、跨地区汇聚融合和深度利用，打破技术和协议壁垒，
实现数据互联互通互操作[7]，这使得高效、安全的跨域数据交换技术成为当前
信息化建设的重点方向。
1.1.3 传统点对点数据交换方式的局限性
　　传统的数据交换主要采用点对点接口方式，大部分基于Web Service、RPC
或专用API的直接对接。这种方式在早期小规模场景中较为有效，但在跨域环
境下暴露出一系列严重局限：最明显的问题是，耦合度高--数据生产者和消费
者需相互知晓接口细节，一旦新增节点或变更交互规则，就需对全链路系统进行
大规模改造，维护成本极高。这同时也导致了扩展性差，N 个系统需维护N×
(N-1)/2个接口，参与方增多时接口数量爆炸式增长，难以支持多对多动态订阅
场景；其次，在跨域场景下涉及网络隔离、安全域划分、数据隐私保护等复杂问
题，直接对接容易引发数据泄露或系统入侵风险，进而导致敏感信息被非法访问。
异构数据格式转换与网络波动进一步加剧不可靠性，已难以满足当前大规模、实
时性强的跨域数据共享需求。
1.1.4 发布-订阅机制的天然优势
　　发布-订阅（Publish/Subscribe，简称Pub/Sub）机制是一种典型的异步解耦
消息传递模式，其核心在于通过主题（Topic）实现生产者与消费者的间接交互：
数据生产者仅需将消息发布至指定主题，而消费者通过订阅感兴趣的主题即可获
取数据，双方无需知晓彼此存在。这种模式天然适用于多对多、动态扩展的通信
场景，具有高度的松耦合性和灵活性。在跨域数据交换中，Pub/Sub机制可通过
中间代理（Broker）实现域间消息路由与策略控制，支持"一处发布、多处订阅"，
有效降低直接耦合带来的安全风险，显著降低了点对点直接交互带来的安全隐患。
此外，结合QoS保障与主题级权限管理，Pub/Sub能够有效应对跨域环境中的网
络波动与隐私问题，在大规模分布式系统中展现出更强的扩展性与灵活性。相比
传统点对点方式，Pub/Sub在扩展性、实时性和安全性方面都展现出显著优势，
已在物联网、分布式系统等领域得到广泛验证，为解决跨域数据交换难题提供了
全新的技术路径。

1.2 研究意义
　　理论意义：为跨域数据交换提供一种异步解耦、安全可控的方法，丰富发布
-订阅机制在跨域场景下的理论应用
　　本研究将发布-订阅机制引入跨域数据交换场景，提出了一种基于异步解耦
的消息传递方法，突破了传统点对点接口紧耦合的局限。通过在不同安全域之间
构建代理体系与主题路由机制，实现了生产者与消费者的完全解耦，同时结合访
问控制列表（ACL）、传输层安全（TLS）加密以及服务质量（QoS）保障，形
成了安全可控的跨域数据流动模型。不仅拓展了发布-订阅机制的理论边界，使
其从传统的域内分布式系统应用延伸至网络隔离、权限严格的跨域环境，还为消
息中间件在异构、安全敏感场景下的设计提供了新的理论参考。
　　现实意义：契合当前数据要素流通、隐私计算与安全可信流通的国家政策与
行业发展趋势，具有较强的工程实践价值
　　当前，我国正大力推动数据要素市场化配置与安全可信流通。《"十四五"
数字经济发展规划》《数据安全法》等政策文件明确提出要破除数据壁垒、促进
数据要素有序流动，同时强调隐私保护与安全可控。本研究提出的基于发布-订
阅的跨域交换方法，通过主题级权限控制、端到端加密以及消息持久化等机制，
在实现高效数据流通的同时保障了隐私安全与传输可靠性，高度契合国家在数据
要素流通、隐私计算与零信任架构方面的战略导向。同时，该方法技术门槛适中、
实现成本较低，便于在政务云、企业私有云等实际环境中快速部署与迭代，具有
较强的工程实践价值。通过研究与原型验证，可为行业提供可复制的技术路径，
助力数字经济高质量发展与数据安全治理体系建设。
1.3 国内外研究现状
1.3.1 发布-订阅机制研究现状
　　发布-订阅（Publish/Subscribe，简称Pub/Sub）机制作为一种异步、松耦合
的消息传递范式，已成为分布式系统和物联网领域的核心技术。经典协议与平台
主要包括MQTT、Apache Kafka和AMQP。
　　MQTT（Message Queuing Telemetry Transport）是一种轻量级协议，由IBM
和Eurotech于1999年提出，现为OASIS标准。其特点是基于TCP/IP的主题订
阅机制，支持三种服务质量（QoS）级别（至少一次、至多一次、恰好一次），
适用于资源受限设备和低带宽、不稳定网络环境，主要用于物联网场景，如传感
器数据采集和远程监控。作为物联网领域应用最广泛的应用层协议之一，其轻量
级设计与TCP/IP适配性也得到了相关学术研究的验证[8]。
　　Apache Kafka则是一种分布式事件流平台，以高吞吐量、持久化日志和水平
扩展著称，支持实时数据管道和流处理，常用于大数据分析和日志聚合。
　　AMQP（Advanced Message Queuing Protocol）是一种更丰富的开放标准，以
RabbitMQ为典型实现，支持多种消息模式（包括Pub/Sub、点对点、请求-响应），
提供先进的路由、事务和安全特性，适用于企业级复杂消息系统。

　　在应用方面，发布/订阅（Pub/Sub）机制已在物联网和分布式系统中得到成
熟广泛的应用。MQTT协议广泛应用于智能家居、工业物联网（IIoT）和车联网
（IoV）等领域，其主流经纪人如HiveMQ和EMQX已验证支持数百万至亿级并
发连接；Kafka在实时事件流处理领域占据主导地位，被Netflix、Uber等公司大
规模用于微服务解耦、实时分析和数据湖构建；AMQP协议及RabbitMQ实现则
常用于金融、电信等对消息可靠传递、复杂路由和事务性要求较高的场景。
　　然而，现有Pub/Sub系统在跨域场景下仍存在明显不足：安全隔离机制较弱，
易受域间网络攻击影响；权限控制虽有ACL支持，但细粒度不足，难以实现动
态跨域授权，甚至无法抵御恶意代理与发布者/订阅者的合谋攻击[2]；在弱网或
高延迟跨域环境中，消息持久化和重试策略仍有待提升，容易导致数据丢失或延
迟过高。这些挑战限制了其在安全敏感跨域数据交换中的直接应用[9]。
1.3.2 跨域数据交换技术研究现状
　　跨域数据交换技术旨在解决不同安全域、网络域或组织间的数据隔离与共享
问题，国内外研究呈现从传统硬件隔离向新兴隐私保护的演进。
　　传统技术路径主要包括网闸、前置机、API网关和数据摆渡等。在国内政务
领域，这些技术广泛应用：网闸实现单向或受控双向数据传输，确保高安全域向
低安全域的物理隔离，但传输效率低、实时性差；前置机作为数据交换前端，支
持格式转换和审计，但依赖专用硬件，扩展性有限；API网关（如Spring Cloud 
Gateway）提供服务暴露与流量控制，但耦合度高，易引入安全漏洞；数据摆渡
则通过手动介质或自动化脚本实现离线交换，适用于极端隔离场景，但缺乏实时
性和自动化。这些传统路径的特点是安全优先，但局限在于扩展性差、维护成本
高、难以支持多对多动态交互。
　　新兴技术路径聚焦隐私保护与可信流通，包括零信任架构、隐私计算、联邦
学习和数据沙箱。零信任架构（Zero Trust）强调"永不信任、持续验证"，通过微
分段和动态访问控制实现跨域最小权限访问，已在Google BeyondCorp和国内政
务云中应用，Liu（2024）进一步探索了云-边-端协同环境下的跨域共享 [2]，证
明了结合零信任模型与分布式账本技术可以有效提升跨域数据交换的安全性与
可扩展性；隐私计算允许数据"可用不可见"，支撑跨域联合分析；联邦学习
（Federated Learning）由Google于2016年提出，其核心思想是通过模型参数交
换替代原始数据共享，实现跨设备/机构的协作建模。近年来，该技术在医疗、
金融等领域取得显著进展。Yang（2019）进一步系统化联邦学习框架，提出横向
联邦学习与纵向联邦学习两种范式，通过基于安全聚合（Secure Aggregation）的
加密参数聚合机制，实现"数据不动模型动"的新型协作模式。核心在于："分离
模型训练与数据存储，参与者仅交互加密后的梯度更新，无需暴露原始数据[10]"；
数据沙箱技术则通过提供隔离执行环境，支持数据可用不可见的可控访问，为隐
私保护提供补充方案。
　　基于消息队列的跨域方案探索，目前主要以MQTT与Kafka的桥接机制为
主。例如，通过MQTT-to-Kafka桥接（如EMQX或HiveMQ企业版内置桥接）
可实现物联网数据向大数据平台的实时流转，或借助Kafka Connect及其MQTT 
Connector完成域间数据同步。然而，此类方案在安全隔离层面仍存在显著局限
性： 
　　（1）桥接过程缺乏原生的跨域身份认证与互认机制，难以实现端到端信任
传递，易受中间人攻击； 
　　（2）权限控制难以在跨域场景下实现细粒度的统一管理和动态延伸，主题
级别的ACL难以自动映射； 
　　（3）在网络隔离或单向传输环境中，消息路由与持久化易受网络中断影响，
导致数据一致性和系统可靠性下降。
1.3.3 研究现状综述与本文切入点
　　综上所述，国内外在发布-订阅机制与跨域数据交换领域已取得诸多显著进
展：Pub/Sub协议在域内场景成熟高效，跨域交换技术正从传统硬件隔离向隐私
计算范式转型。然而，现有方案普遍面临耦合度偏高、安全性与扩展性难以两全
的困境--传统路径虽安全但灵活性不足，新兴路径计算复杂度高且部署门槛苛
刻，基于消息队列的方案则在跨域隔离、细粒度权限管控及弱网鲁棒性上仍存在
显著短板，难以满足大规模异构、安全敏感场景的需求。
　　本文的核心切入点是将轻量级Pub/Sub机制（以MQTT为核心）引入跨域
数据交换场景，通过构建代理体系、增强ACL/TLS安全机制及优化消息持久化
策略，打造一种异步解耦、安全可控的改进模型。这一方向不仅能有效弥补现有
方案的短板，还可实现"一处发布、多处订阅"的高效数据流通，具备较强的创
新性与工程实践价值。
1.4 本文主要研究内容与创新点
1.4.1 主要研究内容
　　本文围绕"基于发布-订阅机制的数据跨域交换方法"展开研究，旨在设计并
实现一种高效、安全、可扩展的跨域数据共享方案。主要研究内容包括以下四个
方面：
　　跨域数据交换需求分析与问题建模。首先分析典型跨域场景（如政务协同、
企业供应链、医疗信息共享）中的数据孤岛成因、交换痛点以及安全隔离需求，
明确功能需求（数据发布、订阅、路由、权限控制）和非功能需求（实时性、安

　　基于发布-订阅模式的跨域消息交换模型设计。提出以MQTT Broker为核心
的代理体系架构，在确保域间网络隔离性的基础上，高效实现消息的跨域路由与
分发。核心机制包括：主题管理与路由机制，支持跨域主题的灵活定义与域间高
效转发；安全控制机制，涵盖订阅权限精细化管理、身份双向验证及端到端数据
加密--针对MQTT协议的固有安全漏洞，Laghari（2024）已系统论证通过访问
控制列表（ACL）与加密传输加固协议生态的可行性[3]，为本研究构建安全隔离
的消息通道提供了重要技术支撑；可靠性保障机制，包含消息持久化存储、服务
质量（QoS）分级保障及确认重试策略，确保跨域传输在网络波动场景下的消息
不丢失、不重复。
　　关键技术研究。主要针对跨域场景的核心难点，开展以下工作：研究跨域认
证与授权方法，设计结合ACL与JWT的访问控制方案；实现异构系统间的数据
格式转换，确保消息的规范化处理；研究可靠传输与弱网优化策略，包括消息队
列缓冲、智能重试机制，以及利用Linux Traffic Control（tc）工具模拟高延迟、
高丢包环境进行性能调优，提高系统在弱网条件下的稳定性。
　　原型系统实现与验证。采用MQTT协议（选用开源的EMQX作为Broker）、
Spring Boot 3后端框架和React前端技术栈，搭建一个轻量级的可演示原型系统。
该系统支持多域模拟（通过Docker容器实现网络隔离）、完整的发布/订阅流程
演示以及实时监控功能。验证部分包括功能测试、性能测试和弱网环境下的稳定
性测试，通过采集延迟、丢包率、成功率等实验数据并进行对比分析，评估所提
方法的有效性和可行性。

第二章 相关技术与理论基础
2.1 发布-订阅机制概述核心理论基础
　　发布 - 订阅（Publish/Subscribe，简称 Pub/Sub）是分布式系统中经典的异
步消息传递范式，其核心是通过中间代理节点实现消息生产者与消费者的完全解
耦，解决了传统同步通信模式在大规模分布式场景下的扩展性与灵活性瓶颈。本
章节将从理论范式、核心特性与场景适配性三个维度，为本文的跨域数据交换方
法构建理论基础。
2.1.1 发布 - 订阅范式的核心定义与经典分类
　　发布 - 订阅范式的核心定义为：消息的发送者（生产者 Publisher）无需将
消息直接发送给特定的接收者（消费者 Subscriber），而是将携带分类标签的消
息发布至中间代理（Broker）；消费者通过向 Broker 注册自身感兴趣的标签，
仅接收匹配的消息，生产者与消费者全程无需知晓对方的网络地址、接口规范与
运行状态。该范式的奠基性研究《The many faces of publish/subscribe》将其按路
由模型分为三类核心实现，不同模型的特性与适用场景如下：
1.	基于主题（Topic-based）的路由模型：消息通过层级化的主题字符串进行
分类，Broker 仅根据主题的精确匹配或通配符匹配完成消息路由。该模型
实现轻量、路由效率高、权限控制粒度清晰，是目前工业界应用最广泛的实
现，也是本文采用的核心路由模型。
2.	基于内容（Content-based）的路由模型：Broker 根据消息体的内容属性与
消费者的过滤规则完成匹配路由。该模型灵活性极强，但路由计算开销大、
权限管控难度高，不适合跨域安全隔离场景。
3.	基于类型（Type-based）的路由模型：以消息的类型体系为匹配依据，仅支
持面向对象类型的精确 / 继承匹配，适用场景高度受限，无法适配跨域异
构数据的灵活交换需求。
2.1.2 发布 - 订阅机制的核心解耦理论
　　发布 - 订阅范式的核心价值在于实现了分布式通信的三重解耦，这也是其
能够解决跨域数据交换核心痛点的理论根基：
1.	空间解耦：生产者与消费者无需知晓对方的网络地址、身份信息与部署位置，
仅需与 Broker 建立连接即可完成通信。该特性完美适配跨域场景中不同安
全域的网络隔离要求 -- 跨域交互的双方无需暴露内网服务地址，彻底避
免了点对点直连带来的网络攻击风险。
2.	时间解耦：生产者与消费者无需同时在线即可完成消息传递，Broker 可通
过消息持久化机制缓存离线消息，待消费者上线后完成补发。该特性解决了
跨域场景中网络波动、节点临时离线导致的消息丢失问题，为不稳定网络环
境下的可靠传输提供了理论支撑。
3.	同步解耦：生产者发布消息后无需阻塞等待消费者的响应，可继续执行后续
业务；消费者通过异步回调的方式处理消息，双方的运行流程完全解耦。该
特性大幅降低了跨域交互的链路耦合度，避免了单节点故障对全链路的影响，
提升了系统的可扩展性。
2.1.3 跨域数据交换场景的适配性与核心挑战
　　基于三重解耦的核心特性，发布 - 订阅范式天然适配跨域数据交换 "多对
多动态交互、安全域隔离、网络环境异构" 的核心需求，可有效解决传统点对点
模式的接口爆炸、耦合度高、扩展性差等问题。但现有理论与实现应用于跨域场
景时，仍存在三大核心理论与工程挑战：
1.	跨域场景下的身份互认与细粒度权限传递机制缺失，单域内的访问控制规则
无法安全、动态地延伸至跨域交互场景；
2.	现有消息可靠传输机制针对单域稳定网络设计，在跨域高延迟、高丢包的弱
网环境中，存在消息重传效率低、交付成功率下降的问题；
3.	多 Broker 级联的跨域路由模型设计不足，无法在保障域间网络隔离的前提
下，实现高效的跨域主题消息转发。
本文的研究正是针对上述三大挑战，基于主题路由的发布 - 订阅范式，设计
适配跨域场景的安全、可靠、可扩展的数据交换方法。
2.2 MQTT 协议原理与跨域场景适配性
　　MQTT（Message Queuing Telemetry Transport）是基于主题路由模型的轻量
级发布 - 订阅消息传输协议，由 IBM 与 Eurotech 于 1999 年提出，目前已成
为 OASIS 国际标准与 ISO/IEC 20922 国际标准，专为低带宽、不稳定网络、
资源受限的分布式场景设计。本章节将重点解析其核心原理，并针对跨域数据交
换场景完成适配性分析，为本文的系统设计提供协议层支撑。
2.2.1 MQTT 协议核心架构与规范基础
　　MQTT 协议采用经典的客户端 - 服务器（Client-Broker）架构，无中心化
节点依赖，整体架构分为两个核心角色：
*	MQTT 客户端：可同时承担消息生产者、消费者的双重角色，负责向 
Broker 发布消息、向 Broker 订阅主题并接收匹配的消息。客户端支持
从嵌入式设备到云端服务的全场景部署，适配跨域场景中异构系统的接
入需求。
*	MQTT Broker：协议的核心中间节点，负责客户端连接管理、身份认证、
消息路由、订阅关系维护、消息持久化与 QoS 保障，是实现跨域消息
交换的核心枢纽。主流开源实现（如 EMQX、HiveMQ）已验证可支持
百万级并发连接，满足大规模跨域节点的接入需求。
　　协议固定头部仅 2 字节，协议栈开销极低，基于 TCP/IP 实现可靠传输，
同时支持 TLS 1.3 端到端加密与 QUIC 传输层扩展，完美适配跨域场景对轻量
级、高安全、弱网适配的核心要求。De Rango 等（2025）提出了一种名为
DLST-MQTT的动态轻量级端到端安全机制，通过在发布者和订阅者之间协商安
全密钥并避免Broker解密重加密，显著降低了跨域场景下的计算开销与延迟，
同时保持了主题级隔离的安全性，为本文的端到端加密策略提供了重要参考[12]。
　　
　　
2.2.2 核心功能特性与跨域场景适配性
　　本小节将针对 MQTT 的核心特性，结合跨域数据交换的场景需求完成适配
性分析，并明确原生协议在跨域场景中的优化空间，为本文的创新设计提供铺垫。
1.	层次化主题结构与路由机制
　　MQTT 采用 UTF-8 编码的层级化主题结构，通过/分隔主题层级（如
gov/domain_a/civil_affairs/population），支持单层通配符+与多层通配符#实现灵
活订阅。
*	跨域适配性：可通过主题前缀实现安全域的隔离与划分，为每个独立安
全域分配专属的主题前缀，天然实现跨域消息的路由隔离与权限边界划
分；同时支持 "一处发布、多处订阅" 的跨域多对多数据共享，彻底解
决点对点模式的接口爆炸问题。
*	原生局限：原生主题路由仅支持单 Broker 内的匹配转发，无标准化的
跨 Broker 跨域主题映射与路由机制，需针对跨域场景完成定制化设计。
2.	分级 QoS 服务质量保障
　　MQTT 原生定义了三级可配置的服务质量级别，可根据跨域数据的重要性
灵活选择，为不同场景提供差异化的可靠传输保障：
*	QoS 0（至多一次）：消息仅发送一次，无确认机制，传输开销最低，
适用于非关键的高频状态数据；
*	QoS 1（至少一次）：通过 PUBLISH-PUBACK 两次握手实现消息确认，
未收到确认时触发重传，确保消息至少到达一次，适用于绝大多数跨域
业务数据交换；
*	QoS 2（恰好一次）：通过 PUBLISH-PUBREC-PUBREL-PUBCOMP 四
次握手实现去重与确认，确保消息不丢失、不重复，适用于金融、医疗
等关键敏感数据的跨域交换。
*	跨域适配性：原生 QoS 机制直接针对不稳定网络环境设计，可有效应
对跨域场景中的高延迟、高丢包问题，为跨域可靠传输提供了协议层基
础。
*	原生局限：原生重传策略为固定周期触发，未针对跨域弱网环境做自适
应优化，在长链路中断场景下易出现消息堆积与重传风暴，需进一步优
化。
3.	会话与状态管理机制
　　MQTT 提供持久会话（Clean Session=false）机制，Broker 可持久化存储客
户端的订阅关系与离线消息，待客户端重连后自动补发未交付的消息；同时提供
遗嘱消息（Will Message）机制，客户端异常离线时，Broker 自动向指定主题发
布预设的遗嘱通知，实现节点故障的快速感知。
*	跨域适配性：持久会话机制解决了跨域节点因网络波动离线导致的消息
丢失问题，遗嘱消息机制实现了跨域节点的状态监控，为大规模跨域节
点的运维管理提供了原生支持。
4.	原生安全扩展机制
　　MQTT 原生支持多维度的安全防护能力，包括用户名 / 密码身份认证、
X.509 证书双向认证、TLS 1.3 端到端传输加密，以及主题级 ACL（访问控制
列表）权限管控，可实现对客户端发布 / 订阅权限的精细化控制。
*	跨域适配性：原生安全机制为跨域数据交换的身份认证、数据加密、权
限隔离提供了基础能力，可有效防范中间人攻击、越权访问等跨域安全
风险。
*	原生局限：原生 ACL 仅支持单 Broker 本地静态配置，无法实现跨域
的动态授权与权限传递，需结合 JWT 等技术完成跨域安全机制的增强
设计。
　　针对MQTT协议在跨域场景下易受中间人攻击与权限越界的问题，
Alotaibi 等（2024）提出利用分布式机器学习框架增强协议的安全性，结合
ACL与异常检测机制，能够有效缓解恶意发布/订阅行为带来的风险，进一
步验证了在异构域间加强动态安全防护的可行性[11]。

2.2.3 MQTT over QUIC 传输增强机制
　　针对跨域弱网场景的传输痛点，OASIS MQTT 3.1.1 与 5.0 标准已正式纳入 
MQTT over QUIC 扩展规范。QUIC（Quick UDP Internet Connections）是基于 
UDP 的新一代传输层协议，由 IETF 标准化为 RFC 9000，相比 TCP 协议，其
在跨域场景中具备三大核心优势：
*	0-RTT/1-RTT 快速握手：相比 TCP+TLS 的 3-RTT 握手，QUIC 可实
现 1-RTT 完整握手与 0-RTT 会话复用，大幅降低跨域长链路的连接建
立延迟，同时支持连接迁移，客户端网络切换时无需重建连接，适配跨
域异构网络的接入需求。
*	无队头阻塞的多路复用：QUIC 的多路复用基于流 ID 实现，单条连接
上的多个流相互独立，单个流的丢包不会影响其他流的传输，彻底解决
了 TCP 的队头阻塞问题，在跨域高丢包环境中可显著提升传输效率。
*	前向纠错与拥塞控制优化：原生支持前向纠错（FEC）机制，可通过冗
余包恢复丢包数据，减少重传次数；同时支持可定制的拥塞控制算法，
可针对跨域网络特征完成自适应优化。
　　针对跨域弱网环境的高延迟与丢包问题，MQTT over QUIC通过0-RTT快速
重连、多路复用无队头阻塞以及前向纠错机制，显著提升了消息交付效率与稳定
性，尤其适用于网络不稳定、节点频繁切换的政务与供应链跨域场景[13]。本文
将基于 MQTT over QUIC 扩展，设计跨域弱网环境下的传输优化策略，进一步
提升系统在不稳定网络中的鲁棒性。
2.3 主流发布 - 订阅消息中间件对比与选型论证
　　当前工业界主流的发布 - 订阅消息中间件，除 MQTT 外，以 Apache Kafka 
与 AMQP 协议（典型实现为 RabbitMQ）应用最为广泛。本章节将针对跨域数
据交换的核心需求，设定专项对比维度，完成三类中间件的适配性分析，并给出
本文的选型决策论证。
2.3.1 选型维度设定
　　结合本文跨域数据交换场景的核心需求，本次选型设定 7 个核心对比维度，
所有维度均围绕跨域场景的核心痛点设计，具体如下：
1.	协议轻量级：协议头部开销、资源占用情况，直接影响跨域低带宽链路的传
输效率；
2.	跨域弱网适应性：对高延迟、高丢包、频繁断连的跨域网络的适配能力，包
括原生 QoS 机制、离线消息支持、传输层优化能力；
3.	细粒度权限控制能力：对跨域多安全域隔离的支持能力，包括权限管控粒度、
动态配置能力、跨域授权扩展能力；
4.	跨域部署复杂度：跨隔离域部署的运维成本、集群配置难度、节点间同步开
销；
5.	端到端安全传输支持：原生支持的加密传输、身份认证能力，直接影响跨域
数据传输的安全性；
6.	离线消息持久化能力：对跨域节点离线场景的适配能力，包括持久化开销、
离线消息补发机制；
7.	高并发接入能力：对大规模跨域节点接入的支持能力，包括并发连接数上限、
消息吞吐性能。

2.3.3 跨域场景下的选型决策论证
　　基于上述专项对比，结合本文的研究场景与核心需求，本文最终选择MQTT 
协议（EMQX 开源 Broker 实现） 作为跨域数据交换系统的核心通信引擎，核心
决策依据如下：
1.	核心需求高度匹配：本文针对的政务协同、供应链管理等跨域数据交换场景，
核心需求是轻量部署、弱网高可靠、细粒度安全隔离、低耦合易扩展，而非
极致的大数据吞吐能力。MQTT 在所有核心需求维度均具备显著优势，完美
适配场景定位。
2.	跨域场景原生适配：Kafka 的核心优势在于高吞吐的大数据流处理，其集群
副本同步对网络质量要求极高，跨域高延迟、高丢包环境下易出现集群稳定
性下降、同步失败等问题，无法适配跨域隔离场景；AMQP 协议栈复杂、部
署运维成本高，弱网环境下重传开销大，不适合多隔离域的轻量化部署。而 
MQTT 专为不稳定网络设计，轻量级的协议特性与原生的安全、可靠机制，
天然适配跨域场景的核心痛点。
3.	扩展能力满足创新设计需求：EMQX 作为主流的开源 MQTT Broker，原生支
持 MQTT over QUIC 扩展、ACL 热更新、规则引擎与持久化插件，可无缝对
接本文设计的跨域主题路由、动态授权、弱网优化等创新机制，无需大规模
二次开发即可完成原型系统的构建，具备极强的工程落地性。
2.4 跨域数据交换领域相关技术
　　跨域数据交换技术的核心目标，是在保障不同安全域、网络域、组织域的隔
离边界与数据安全的前提下，打破数据孤岛，实现可控、高效的数据跨域流通。
其技术演进路径整体呈现从 "强物理隔离优先" 的传统方案，向 "安全与灵活性
兼顾" 的可信流通方案逐步迭代的趋势。本章节将围绕跨域数据交换的核心技术
体系，分析不同技术路径的核心逻辑、适用边界与本质瓶颈，明确本文方案在技
术体系中的定位与设计目标。
2.4.1 传统跨域数据交换技术与核心瓶颈
　　传统跨域数据交换技术以 "物理 / 逻辑强隔离" 为核心设计原则，广泛
应用于政务、金融等高安全等级场景，核心实现路径包括网闸、前置机、API 网
关与数据摆渡四类。
1.	网闸技术：通过物理层或链路层的单向 / 受控双向数据传输机制，实现高
安全域与低安全域之间的隔离式数据交换，核心是切断两个域之间的直接网
络连接，通过摆渡芯片完成数据的单向转发。其核心优势是物理隔离强度高，
可有效抵御网络层攻击；但本质瓶颈在于传输效率低、实时性差，仅支持固
定格式的批量数据传输，无法适配多对多的动态数据订阅场景，也难以支持
高频实时的跨域业务协同。
2.	前置机技术：在每个安全域的边界部署专用前置服务器，作为域内系统与域
外系统的唯一交互入口，所有跨域数据交换均通过双方前置机完成，同时承
担数据格式转换、传输审计、安全过滤的功能。该方案实现了域内系统与域
外网络的隔离，但本质仍为点对点的对接模式，N 个跨域交互方需维护 
N×(N-1)/2 个前置机对接链路，新增节点时需全链路改造，扩展性极差，维
护成本随接入方数量呈指数级增长。
3.	API 网关技术：通过统一的网关入口实现服务接口的暴露、流量控制与身份
认证，将域内的业务能力封装为标准化 API 接口供域外调用，是当前应用
最广泛的跨系统对接方案。该方案解决了接口统一管理的问题，但本质仍为
同步请求 - 响应的紧耦合模式，数据生产者与消费者需提前约定接口规范，
无法支持动态的多对多数据共享；同时跨域 API 直接暴露在公网环境中，
易引发越权访问、中间人攻击等安全风险，难以适配多安全域的严格隔离要
求。
4.	数据摆渡技术：通过离线存储介质或自动化脚本，实现两个隔离域之间的非
实时数据同步，分为人工介质摆渡与自动化离线摆渡两类。该方案适用于极
端物理隔离的场景，但核心缺陷是实时性极差，无法满足业务协同对数据时
效性的要求，同时自动化程度低，易出现数据不一致、传输遗漏等问题。
　　综上，传统跨域数据交换技术虽能满足强安全隔离的基础要求，但普遍存在紧耦
合、扩展性差、灵活性不足、实时性弱的本质瓶颈，无法适配当前大规模、动态化、
多对多的跨域数据共享需求，这也是本文研究要解决的核心痛点。
2.4.2 新兴跨域安全可信流通技术
　　随着数据安全相关法规的完善与隐私保护技术的发展，跨域数据交换技术逐
步向 "数据可用不可见" 的可信流通方向演进，核心技术路径包括零信任架构、
隐私计算与联邦学习、数据沙箱三类。
1.	零信任架构：以 "永不信任、持续验证、最小权限" 为核心设计原则，打破
了传统网络边界信任的模式，对每一次跨域访问请求都进行全链路的身份认
证与权限校验，通过微分段技术实现细粒度的访问控制，确保跨域访问仅能
获取完成业务所需的最小数据权限。该架构为跨域数据交换的安全管控提供
了核心设计理念，本文提出的主题级细粒度 ACL 控制、端到端身份认证、
动态授权机制，正是零信任核心原则在发布 - 订阅跨域交换场景中的轻量
化落地。
2.	隐私计算与联邦学习：隐私计算通过同态加密、多方安全计算等密码学技术，
实现数据在加密状态下的计算与分析，达成 "数据可用不可见" 的效果；联
邦学习则是隐私计算的典型应用范式，通过跨域交换加密后的模型梯度参数，
替代原始数据的直接共享，实现跨机构的协同建模。这类技术解决了跨域数
据共享中的隐私泄露问题，但其核心局限在于计算开销大、部署门槛高，对
硬件与网络环境要求严苛，难以适配轻量化、高频次的通用跨域数据交换场
景。本文提出的跨域数据交换方案与隐私计算技术为互补关系，可通过在数
据发布前完成隐私脱敏处理，再通过本文的交换通道实现安全流通，形成 
"隐私处理 + 安全流通" 的完整跨域共享体系。
3.	数据沙箱技术：通过构建隔离的、可控的代码执行环境，将跨域共享的数据
集中存储在沙箱内，数据消费者仅能在沙箱内完成数据的计算与分析，无法
导出原始数据，实现数据的可控访问。该技术适用于跨域联合分析场景，但
无法支持原始数据的跨域流通与业务协同，适用场景高度受限，与本文面向
通用跨域数据交换的设计目标匹配度较低。
2.4.3 基于消息队列的跨域方案现存短板
　　基于发布 - 订阅消息队列的跨域数据交换方案，是当前行业内兼顾灵活性
与安全性的主流探索方向，现有方案多以 MQTT 与 Kafka 的桥接机制为核心，
通过不同域的 Broker 节点桥接，实现跨域的消息同步与转发。但现有方案在跨
域场景的落地中，仍存在三大核心短板，也是本文研究的核心优化方向：
1.	跨域身份互认与权限传递能力缺失：现有桥接方案仅实现了消息的跨域转发，
未建立标准化的跨域端到端身份互认机制，不同域的身份体系无法互通，易
出现中间人攻击与身份伪造风险；同时单域内的主题级 ACL 权限规则无法
跨域动态映射，难以实现细粒度的跨域访问控制，易出现越权订阅、非法发
布等安全问题。
2.	域间隔离边界的安全性不足：现有桥接方案为实现消息转发，需打通不同域 
Broker 节点之间的直接网络连接，打破了原有安全域的网络隔离边界，将
单域内的风险扩散至全链路，一旦单个 Broker 节点被入侵，将引发全域的
安全风险，无法满足高安全等级跨域场景的隔离要求。
3.	弱网环境下的传输可靠性不足：现有方案的消息持久化与重传机制均针对单
域稳定网络设计，未针对跨域高延迟、高丢包、频繁断连的弱网环境做优化，
跨域链路中断时易出现消息丢失、重复转发、数据不一致等问题，在复杂跨
域网络环境中的鲁棒性不足。
2.4.4 本文方案的技术定位
　　综上，现有跨域数据交换技术普遍面临 "安全隔离与灵活扩展难以兼顾" 
的核心困境：传统强隔离方案灵活性不足，无法适配动态多对多的共享需求；新
兴隐私计算方案部署门槛高，难以大规模推广；现有消息队列桥接方案在跨域安
全管控与弱网可靠性上存在明显短板。
　　本文提出的基于发布 - 订阅机制的跨域数据交换方法，正是填补了上述技
术空白：以 MQTT 协议为核心，通过异步解耦的发布 - 订阅模式解决传统方案
的紧耦合问题，通过增强的跨域身份认证与主题级 ACL 控制保障域间隔离与数
据安全，通过优化的持久化与重传机制提升弱网环境下的传输可靠性，最终形成
一套兼顾安全隔离、异步解耦、轻量易部署、弱网高可靠的跨域数据交换方案。
2.5 原型系统开发与验证技术栈
　　为验证本文提出的跨域数据交换方法的有效性与可行性，本研究搭建了可演
示、可测试的原型系统。本章节按照系统分层架构，阐述原型系统的核心技术选
型及其在系统中的核心作用，为后续的系统实现与测试验证提供技术支撑。
　　核心消息引擎技术：原型系统选用 EMQX 开源 MQTT Broker 作为核心消
息引擎，承担跨域消息的路由转发、订阅关系维护、QoS 服务质量保障、访问
控制与消息持久化的核心功能，是整个跨域数据交换系统的数据平面核心。
　　后端管理控制平面开发技术：原型系统的管理控制平面采用 Spring Boot 3 
框架进行开发，承担系统的用户管理、主题管理、ACL 权限配置、跨域路由规
则管理、日志审计、监控数据采集与处理的核心功能，是实现系统可视化管控的
核心载体。
　　前端可视化技术：原型系统的前端管理控制台采用 React 框架结合 ECharts 
可视化库进行开发，核心实现系统的交互管控界面与数据可视化功能，包括用户
登录、主题与权限配置、跨域节点状态监控、消息流量统计、数据流向拓扑展示、
跨域流转轨迹可视化等核心功能。
　　跨域隔离与弱网仿真验证技术：为模拟真实的跨域安全隔离环境与复杂网络
场景，验证系统的隔离性与鲁棒性，原型系统采用 Docker 容器技术与 Linux 
Traffic Control（tc）工具构建测试验证环境。其中，Docker 容器技术通过独立
的网络命名空间，为每个安全域构建相互隔离的容器集群与虚拟网络，不同安全
域之间仅能通过 Broker 节点实现受控的消息交换，用于模拟真实场景中的跨域
网络隔离环境；Docker Compose 可实现多域环境的部署与启停。Linux Traffic 
Control（tc）工具通过其 netem 网络模拟模块，可在 Docker 宿主机的虚拟网
卡上配置网络延迟、丢包率、带宽限制、数据包乱序等参数，模拟跨域场景中的
弱网环境，用于测试系统在复杂网络条件下的消息交付成功率、传输延迟与稳定
性，验证弱网优化策略的有效性。
　　安全增强配套技术：为保障跨域数据传输的安全性，原型系统配套采用 TLS 
1.3 传输加密与 JWT 令牌认证两项核心安全技术。其中，TLS 1.3 协议用于实
现 MQTT 客户端与 Broker 之间、前后端服务之间的端到端传输加密；JWT
（JSON Web Token）技术用于实现无状态的跨域身份认证，结合主题级 ACL 规
则，实现跨域客户端的动态授权与权限校验。
　　
2.6 本章小结
　　本章围绕本文的研究主题，系统梳理了跨域数据交换方法的核心理论基础与
技术支撑体系，完成了从理论范式到技术落地的全链条铺垫。

第三章 系统需求分析与总体设计
3.1 概述

完成系统需求的拆解与形式化定义，明确系统的设计边界与核心目标；在此基础

3.2 业务需求分析
　　本文所设计系统的核心业务目标，是解决不同自治安全域之间的数据可控流
通难题，在保障各安全域网络隔离边界、数据安全与访问权限可控的前提下，打
破传统点对点模式的紧耦合瓶颈，实现 "一处发布、多处订阅" 的灵活跨域数据
共享，适配政务协同、供应链管理、医疗信息共享等典型场景的共性业务诉求。
3.2.1 跨域场景描述
　　本文所指的安全域，是指具备独立网络边界、统一安全策略、自治管理权限
的网络与业务系统集合，典型如政府部门的内网业务域、企业的生产管理域、医
疗机构的患者数据域等。不同安全域之间默认存在严格的网络隔离与访问控制，
无直接的网络互通通道，域内核心业务系统无法直接对外暴露服务接口，这是跨
域数据交换必须遵守的核心约束条件。
　　基于上述约束，本文将跨域数据交换的全链路划分为三个核心逻辑层，数据
从生产域产生后，仅能通过中间层完成受控流转，最终到达授权的消费域，全程
无生产域与消费域的直接交互，具体分层定义如下：
图 2
1.	数据生产域：是跨域交换数据的产生与提供方，域内部署业务系统与授权的
数据生产客户端，负责将待共享的业务数据封装为标准化 MQTT 消息，发
布至指定主题。生产域具备严格的出站访问控制，仅允许授权的生产客户端
与跨域交换核心 Broker 建立单向的加密发布连接，禁止域外节点主动访问
域内系统，从网络层面保障生产域的安全隔离，避免内网暴露带来的攻击风
险。
2.	跨域交换中间层：是整个系统的核心枢纽，基于 MQTT Broker 集群构建，
承担跨域消息的路由转发、身份认证、权限校验、QoS 保障与消息持久化
的核心职责。中间层部署在中立的、可被各安全域授权访问的网络环境中，
与各生产域、消费域之间仅通过加密的 MQTT 协议端口实现受控互通，不
直接处理业务逻辑、不篡改原始数据，仅完成消息的代理转发，彻底解耦生
产域与消费域的直接交互，同时实现跨域交互的全链路审计。
3.	数据消费域：是跨域交换数据的接收与使用方，域内部署授权的数据消费客
户端，通过订阅授权的主题获取对应数据。消费域具备严格的入站访问控制，
仅允许授权的消费客户端与 Broker 建立加密订阅连接，禁止域外节点直接
访问域内系统，保障消费域的内网安全与数据隐私。
　　以政务协同的典型场景为例：民政部门（生产域）需要向人社、医保、公安
等多个平行部门（消费域）共享人口基础信息，传统点对点模式下，民政需分别
与每个部门对接接口、打通网络，维护成本随接入方数量指数级增长；而基于本
系统的架构，民政仅需将标准化后的人口数据发布至
/domain/gov/minzheng/population主题，人社、医保等部门通过订阅该主题即可获
取数据，全程无需暴露各自的内网服务地址，仅通过中间 Broker 完成消息流转，
同时通过 ACL 权限控制确保仅授权部门可订阅该主题，完美适配跨部门政务协
同的业务需求。
3.2.2 发布 - 订阅核心流程
　　结合跨域场景的安全隔离要求，本文定义的跨域发布 - 订阅全流程分为前
置授权流程与核心消息流转流程两部分，完整定义了数据从生产到消费的全链路
逻辑。
前置授权流程
　　跨域场景区别于普通域内发布订阅的核心前提，是必须完成预先的身份认证
与权限授权，避免非法节点接入与越权访问，具体流程如下：
1.	生产域与消费域的客户端向系统管理平台提交接入申请，提交所属安全域、
身份凭证、待发布 / 待订阅的主题范围与业务用途；
2.	管理平台完成合规性审核后，为客户端生成唯一的身份标识与 JWT 访问令
牌，同时在 MQTT Broker 中配置对应的 ACL 规则，明确客户端仅可在指
定主题下执行发布 / 订阅操作，禁止越权访问；
3.	客户端通过 TLS 1.3 加密通道与 Broker 集群建立连接，提交身份凭证完成
双向身份认证，认证通过后维持长连接与心跳机制，确保连接的有效性与链
路安全性。


核心消息流转路径
　　本文定义的跨域消息核心流转路径为：
　　授权数据生产客户端 → TLS 加密 MQTT 连接 → 对应主题 Topic → 
MQTT Broker 核心集群（权限校验、持久化、路由匹配） → TLS 加密 MQTT 
连接 → 所有授权的订阅客户端
　　 
全流程的分步详细逻辑如下：
1.	数据发布流程：生产域的授权客户端将待共享的业务数据进行标准化封装，
添加数据来源、时间戳、数据类型等元数据，生成符合 MQTT 协议规范的
消息体，指定对应的主题与 QoS 等级，通过预先建立的 TLS 加密连接发
送至 Broker 集群；Broker 接收到消息后，首先校验该客户端是否具备该主
题的发布权限，校验不通过则直接丢弃消息并记录违规审计日志，校验通过
则进入后续路由流程。
2.	订阅注册流程：消费域的授权客户端在连接建立后，向 Broker 发送订阅请
求，指定待订阅的主题（支持单层通配符+与多层通配符#匹配）与接收消息
的 QoS 等级；Broker 接收到订阅请求后，校验该客户端是否具备该主题的
订阅权限，校验通过则将该客户端加入对应主题的订阅列表，同时向客户端
返回订阅确认响应；若校验不通过则拒绝订阅请求并记录审计日志。
3.	消息路由与交付流程：Broker 对通过权限校验的发布消息，根据主题匹配
对应的订阅列表，将消息转发至列表中所有在线的授权订阅客户端；针对离
线的订阅客户端，若客户端开启了持久会话机制，Broker 将消息持久化至
本地存储，待客户端重新上线后自动补发，确保消息不丢失；同时根据消息
指定的 QoS 等级，完成对应的握手确认与超时重传机制，保障跨域弱网环
境下的传输可靠性。
4.	全链路审计流程：整个发布 - 订阅流程的所有操作，包括客户端连接、身
份认证、发布请求、订阅请求、消息转发、权限拒绝等，均会被同步记录至
系统审计日志，支持后续的操作追溯、合规审计与异常排查，满足跨域数据
交换的监管与合规性要求。
　　
3.3 系统功能需求分析
　　基于前文的跨域业务场景与核心流程，本系统的功能需求围绕 "安全可控的
跨域发布 - 订阅数据交换" 核心目标，拆解为连接管理、消息交换、传输保障、
系统监控与审计四大核心功能模块，覆盖从客户端接入到消息全生命周期流转、
系统运维的全流程需求。
　　图 3
3.3.1 连接管理功能
　　连接管理是跨域系统的第一道安全防线，核心目标是实现客户端的合法接入、
连接状态的全生命周期管控，确保仅授权节点可接入系统，具体包含两个核心子
功能：
1.	身份认证与权限校验：系统支持基于 X.509 证书、用户名 / 密码结合 JWT 
令牌的多模式身份认证，对接 ACL 访问控制列表。客户端发起连接请求时，
系统首先校验身份凭证的合法性，再匹配预配置的 ACL 规则，明确该客户
端可操作的主题范围、可执行的动作（发布 / 订阅）；对身份非法、权限
越界的连接请求直接拒绝，并记录违规审计日志。针对跨域多安全域的场景，
支持按安全域划分身份体系，实现域内身份的统一管理与跨域互认。
2.	连接心跳与状态维持：系统基于 MQTT 协议的 PINGREQ/PINGRESP 心
跳机制，实现客户端连接状态的实时监控。客户端与 Broker 建立连接后，
需按协商的心跳周期发送心跳包；若系统在约定超时时间内未收到心跳包，
将判定客户端离线，自动清理非持久会话的会话状态，或为持久会话保留离
线消息，同时触发预设的遗嘱消息通知，确保连接状态的一致性，避免无效
连接占用系统资源，为跨域节点的故障感知提供支撑。
3.3.2 消息交换功能
　　消息交换是系统的核心业务功能，基于发布 - 订阅范式实现跨域数据的可
控流转，核心目标是为数据生产端与消费端提供灵活、低耦合的消息交互能力，
具体包含三个核心子功能：
1.	主题管理与发布功能：系统支持层级化的主题命名与管理，可按安全域、业
务类型、数据类型划分主题层级（如/gov/{domain}/{business}/{data_type}），
实现跨域主题的隔离与规范管理。授权的生产客户端可向指定主题发布符合 
MQTT 协议规范的消息，支持自定义消息的 QoS 等级、保留消息标记、过
期时间等属性；系统接收到发布请求后，先校验客户端的主题发布权限，校
验通过后执行消息持久化与路由转发。
2.	主题订阅与消息分发功能：授权的消费客户端可向系统发起主题订阅请求，
支持精确主题匹配、单层通配符（+）与多层通配符（#）的模糊匹配，满足
多主题批量订阅的需求。系统接收到订阅请求后，校验客户端的主题订阅权
限，校验通过后将该客户端加入对应主题的订阅列表，后续该主题下的新消
息将自动转发至所有授权的订阅客户端。同时支持客户端发起取消订阅请求，
系统将同步更新订阅列表，停止对应主题的消息转发。
3.	消息生命周期管理：系统支持对消息的全生命周期管控，包括消息的持久化
存储、过期自动清理、保留消息管理、死信消息处理。针对超过有效期的消
息、无法交付的死信消息，系统自动执行清理或转入死信队列，避免无效消
息占用存储资源；同时支持消息的回溯查询，满足合规审计的需求。
3.3.3 传输保障功能
　　传输保障功能是跨域数据可靠交换的核心支撑，核心目标是应对跨域网络的
不稳定性，为不同重要等级的数据提供差异化的可靠传输保障，具体基于 MQTT 
协议的三级 QoS 机制实现，分为三个等级的传输保障能力：
1.	QoS 0 至多一次传输：适用于非关键的高频状态类数据（如设备运行状态、
实时监控采样数据），消息仅发送一次，系统不执行确认与重传机制，传输
开销最低、延迟最小，可满足高频低优先级数据的跨域传输需求。
2.	QoS 1 至少一次传输：适用于绝大多数业务类跨域数据交换场景（如政务协
同数据、供应链业务数据），通过 PUBLISH-PUBACK 两次握手实现消息
确认机制。系统发布消息后启动超时计时器，若在超时时间内未收到接收方
的确认响应，将触发消息重传，确保消息至少成功交付一次；同时通过消息 
ID 去重机制，避免消费端重复处理消息。
3.	QoS 2 恰好一次传输：适用于金融、医疗等关键敏感数据的跨域交换场景，
通过 PUBLISH-PUBREC-PUBREL-PUBCOMP 四次握手实现严格的去重
与确认机制，确保消息在网络波动、重传的场景下，仅被消费端处理恰好一
次，彻底避免消息丢失与重复处理的风险，为高敏感跨域数据提供最高等级
的传输保障。
3.3.4 系统监控与审计功能
　　系统监控与审计功能是保障跨域系统可观测、可运维、可追溯的核心支撑，
核心目标是实现系统运行状态的实时可视化监控与全链路操作的合规审计，具体
包含两个核心子功能：
1.	实时可视化监控：基于 React 与 ECharts 框架实现系统运行状态的可视化
监控，核心监控指标包括：跨域消息交换频率（发布 / 订阅消息的 TPS）、
消息传输平均延迟、在线客户端数量与分域分布、主题数量与订阅关系拓扑、
系统资源占用率（CPU、内存、网络带宽）。支持监控指标的实时刷新、历
史趋势查询、异常阈值告警，为系统运维人员提供直观的运行状态视图，及
时发现与定位系统异常。
2.	全链路操作审计：系统对全流程的所有操作行为进行全量审计日志记录，包
括客户端连接 / 断开、身份认证结果、发布 / 订阅请求、权限校验结果、
消息转发操作、异常行为等。审计日志包含操作主体、操作时间、操作类型、
操作内容、操作结果、客户端 IP、所属安全域等全维度信息，支持日志的
多条件查询、回溯与导出，满足跨域数据交换的合规监管要求，同时为异常
行为排查、安全事件追溯提供数据支撑。
3.4 系统非功能需求分析
　　非功能需求是系统在跨域复杂场景下稳定、安全、高效运行的核心保障。针
对跨域数据交换场景的网络异构、安全隔离要求高、节点规模动态变化等核心特
点，本系统的非功能需求重点围绕安全性、鲁棒性与可靠性、可扩展性三大核心
维度展开，确保系统可适配真实跨域场景的复杂运行环境。
3.4.1 安全性需求
　　安全性是跨域数据交换系统的首要非功能需求，由于系统涉及不同安全域的
敏感数据流通，面临身份伪造、中间人攻击、越权访问、数据泄露等多重安全风
险，核心需求如下：
1.	端到端传输加密需求：系统必须支持 TLS 1.3 协议，实现客户端与 Broker 
之间、Broker 节点之间的端到端传输加密，确保跨域传输的消息在网络链
路中全程处于加密状态，无法被窃听、篡改或伪造，有效防范中间人攻击。
针对跨域弱网场景，同步支持 MQTT over QUIC 的加密传输能力，在保障
传输安全性的同时，兼顾弱网环境下的传输效率。系统强制所有跨域连接必
须采用加密传输通道，禁止客户端以明文方式接入，杜绝明文传输带来的数
据泄露风险。
2.	细粒度访问控制需求：系统必须实现主题级的细粒度 ACL 访问控制，支持
按客户端身份、所属安全域、操作类型（发布 / 订阅）、主题范围四个维
度配置权限规则，可实现 "某安全域的某客户端仅可向指定主题发布消息、
仅可订阅指定范围的主题" 的精细化管控，禁止越权发布、越权订阅的行为。
同时支持权限规则的动态热更新，无需重启系统即可完成跨域权限的调整，
适配跨域场景中节点权限动态变化的需求。
3.	非法客户端与异常行为拦截需求：系统必须具备非法接入拦截与异常行为检
测能力，针对身份凭证无效、权限校验失败的连接请求，直接拒绝接入并记
录黑名单；对短时间内多次发起非法连接请求的 IP 地址，执行临时封禁策
略。同时针对异常行为（如高频发布超大消息、批量订阅无权限主题、异常
心跳频率），系统可自动识别并执行告警、限流、断开连接等处置操作，防
范恶意客户端对系统的攻击与资源占用，保障跨域系统的稳定运行。
4.	数据安全与合规审计需求：系统必须实现跨域数据交换的全链路可追溯，所
有消息的发布、转发、订阅全流程均需留存可审计的记录；同时支持对敏感
数据的发布前脱敏处理能力，可针对身份证号、手机号等敏感字段自动执行
脱敏，避免敏感数据的跨域泄露。审计日志不可篡改、不可删除，留存周期
需满足相关行业合规要求，为监管审计、安全事件追溯提供完整支撑。
3.4.2 鲁棒性与可靠性需求
　　跨域场景中，不同安全域之间的网络链路普遍存在高延迟、高丢包、频繁断
连等弱网特征，同时系统需支撑 7×24 小时不间断的业务运行，鲁棒性与可靠
性是系统适配真实跨域场景的核心能力，核心需求如下：
1.	弱网环境下的可靠传输需求：系统必须具备应对跨域弱网环境的能力，针对
高延迟、高丢包、频繁断连的网络场景，在 MQTT 原生 QoS 机制的基础
上，优化消息持久化与智能重试策略。具体要求包括：支持消息的本地持久
化存储，消息发布后先持久化到磁盘，再执行转发，避免系统宕机导致的消
息丢失；针对跨域链路中断的场景，支持智能指数退避重试机制，避免固定
周期重试带来的网络拥塞；同时支持离线消息的持久化缓存，待客户端重新
上线后自动补发。系统需满足在网络丢包率高达 20% 的弱网环境下，QoS 
1 与 QoS 2 等级的消息交付成功率不低于 99.99%。
2.	故障容错与高可用需求：系统必须支持集群化部署，核心 Broker 节点采用
无状态集群架构，单个节点故障不影响整个集群的正常运行，可自动实现故
障转移与请求负载均衡，避免单点故障导致的系统瘫痪。同时支持消息的多
副本持久化存储，单节点存储故障时，可从其他副本恢复消息数据，确保消
息不丢失。系统整体可用性需达到 99.95% 以上，年非计划停机时间不超过 
4.38 小时，满足政务、企业级业务的连续运行需求。
3.	流量控制与过载保护需求：系统必须具备流量控制与过载保护能力，针对跨
域场景中突发的高流量消息发布请求，可自动执行限流策略，按客户端、安
全域、主题三个维度设置消息发布的流量阈值，避免突发流量打满系统资源。
同时支持消息体大小限制、连接数限制，防止超大消息、过量连接占用系统
带宽与内存资源，确保系统在高负载场景下仍可稳定运行，不会出现宕机或
服务不可用的情况。
3.4.3 可扩展性需求
　　跨域数据交换场景中，接入的安全域数量、客户端节点规模、业务主题数量
均会随业务发展动态增长，系统必须具备优秀的可扩展性，可随业务需求灵活扩
展，核心需求如下：
1.	容器化部署与横向扩展能力：系统必须支持基于 Docker 容器技术的容器化
部署，所有核心模块（Broker 集群、后端管理服务、前端控制台）均提供
标准化的 Docker 镜像与 Docker Compose 部署脚本，可实现跨环境的一键
部署与启停，大幅降低部署与运维成本。同时核心 Broker 集群支持水平横
向扩展，可通过增加集群节点数量，线性提升系统的并发连接承载能力与消
息吞吐能力，可支撑从数十个节点的小规模跨域场景，到数万个节点的大规
模跨域业务场景，适配业务规模的动态增长。
2.	功能模块化与可扩展能力：系统采用分层模块化的架构设计，核心功能模块
（连接管理、权限控制、消息路由、监控审计）均采用低耦合的模块化设计，
模块之间通过标准化接口交互，可针对业务需求灵活新增、替换功能模块，
如新增隐私计算插件、数据格式转换插件、第三方身份系统对接插件等，无
需修改系统核心架构，具备极强的功能扩展能力，可适配不同行业跨域场景
的个性化需求。
3.	多协议与异构系统适配能力：系统必须具备良好的异构系统适配能力，除核
心的 MQTT 协议外，支持通过协议网关适配 HTTP、WebSocket、AMQP 等
主流协议，可实现不同协议的异构系统之间的跨域数据交换，无需改造原有
业务系统的协议架构，降低系统的接入门槛。同时支持多种数据格式的自动
转换能力，可适配 JSON、XML、二进制等不同格式的消息数据，满足异
构系统之间的跨域数据互通需求。
3.4.4 其他非功能需求
　　除上述核心维度外，系统还需满足基础的性能与易用性需求：
*	性能需求：单 Broker 节点需可支撑万级并发客户端连接，单节点消息转发
平均延迟较低，满足大规模跨域节点的接入与消息流转需求；
*	易用性需求：提供可视化的 Web 管理控制台，支持图形化的权限配置、主
题管理、监控查看、日志审计，降低运维人员的操作门槛，无需掌握底层协
议细节即可完成系统的日常运维管理。
3.5 系统总体架构设计
　　系统总体架构设计遵循分层解耦、安全隔离、高可用、可扩展的核心原则，
针对跨域数据交换场景的网络隔离、弱网传输、细粒度权限管控等核心痛点，采
用经典的分层架构设计，将系统划分为表现层、业务逻辑层、消息引擎层、基础
设施层四个核心层级，各层级之间通过标准化接口交互，实现低耦合、高内聚的
架构设计。系统总体架构图如图 4 所示。
3.5.1 分层架构详细设计
　　（1）表现层
　　表现层是系统与用户交互的可视化入口，核心目标是为系统管理员提供直观、易
用的管控界面，为跨域节点的运维监控提供支撑。
*	核心职责：用户登录与身份认证、主题与权限的图形化配置、跨域节点状态监
控、消息流量与传输延迟的实时可视化、全链路审计日志的查询与导出。
*	核心技术组件：基于 React 框架构建的单页应用（SPA），结合 ECharts 可视
化库实现数据流向拓扑图、消息流量趋势图、在线节点热力图等可视化展示；
采用 Ant Design 组件库提升界面的交互性与美观度。
*	交互逻辑：通过 RESTful API 与业务逻辑层交互，提交管控请求并获取监控数
据与审计日志。
　　（2）业务逻辑层
　　业务逻辑层是系统的管理控制核心，承担业务规则处理与管控逻辑实现的职责，
核心目标是为表现层提供标准化的管控能力，为消息引擎层提供动态配置与权限规
则支撑。
*	核心职责：用户与身份凭证管理、主题层级与路由规则管理、ACL 访问控制规
则的动态配置与下发、跨域节点的状态管理、监控数据的采集与聚合、审计日
志的存储与查询。
*	核心技术组件：基于 Spring Boot 3 框架构建的后端服务，采用 Spring Security 
结合 JWT 实现无状态身份认证；通过 MyBatis Plus 实现数据持久化；集成 
EMQX Java SDK 实现与消息引擎层的深度交互，动态下发 ACL 规则与获取 
Broker 运行状态。
*	交互逻辑：向上为表现层提供 RESTful API，向下通过 EMQX SDK 与消息引
擎层交互，同时将审计日志、监控数据持久化至数据库。
　　（3）消息引擎层
　　消息引擎层是系统的数据平面核心，承担跨域消息的路由转发、QoS 保障、安
全隔离的核心职责，是实现异步解耦跨域数据交换的关键枢纽。
*	核心职责：客户端连接管理与身份认证、主题订阅关系维护、消息路由与转发、
三级 QoS 服务质量保障、消息持久化与离线补发、TLS/QUIC 加密传输支撑。
*	核心技术组件：采用 EMQX 开源 MQTT Broker 集群作为核心消息引擎，单集
群可支撑百万级并发连接；集成 MQTT over QUIC 扩展模块，提升弱网环境下
的传输效率；配置 ACL 插件实现主题级细粒度权限控制；采用集群化部署模
式，通过负载均衡实现高可用与横向扩展。
*	交互逻辑：与数据生产客户端、数据消费客户端通过 MQTT over TLS/QUIC 协
议交互，接收发布请求、订阅请求并转发消息；与业务逻辑层通过 EMQX API 交
互，接收动态配置的 ACL 规则并上报运行状态。
　　（4）基础设施层
　　基础设施层是系统的底层支撑，为上层各层提供运行环境与资源保障，核心目标
是模拟真实的跨域网络隔离环境，支撑系统的容器化部署与弱网测试。
*	核心职责：提供容器化运行环境、构建跨域网络隔离环境、模拟弱网测试环境、
提供数据存储资源。
*	核心技术组件：采用 Docker 容器技术实现各核心模块的容器化部署，通过 
Docker Compose 实现多容器的一键编排；利用 Docker 的网络命名空间构建相
互隔离的虚拟网络，模拟不同安全域的网络隔离；采用 Linux Traffic Control（tc）
工具的 netem 模块，在虚拟网卡上配置网络延迟、丢包率、带宽限制，模拟跨
域弱网环境；采用 MySQL 数据库存储业务数据与审计日志，采用 EMQX 内
置的 RocksDB 存储持久化消息。

3.6 关键技术选型理由
　　第二章已对相关技术的基础原理进行了介绍，本节将结合本系统的跨域数据
交换场景需求，重点阐述各项技术在本系统中的选型理由，说明其如何针对性解
决跨域场景的核心痛点。
3.6.1 MQTT 协议与 EMQX Broker 选型理由
　　本系统选择 MQTT 协议作为核心通信协议、EMQX 作为 Broker 实现，而
非 Kafka 或 RabbitMQ，核心理由如下：
1.	轻量级适配跨域低带宽链路：MQTT 协议固定头部仅 2 字节，协议栈开销
极低，可在跨域低带宽链路中实现高效传输，避免了 Kafka、AMQP 协议
头部开销大的问题；
2.	原生 QoS 机制适配跨域弱网：MQTT 原生支持三级 QoS 服务质量保障，
可针对不同重要性的跨域数据提供差异化的可靠传输，无需二次开发即可应
对跨域网络波动；
3.	EMQX 集群化与扩展能力适配跨域规模：EMQX 单集群可支撑百万级并发连
接，支持集群横向扩展，可适配跨域场景中节点规模动态增长的需求；同时 
EMQX 原生支持 ACL 热更新、规则引擎等扩展功能，可无缝对接本系统
的动态权限配置需求；
4.	主题层级结构适配跨域隔离：MQTT 的层级化主题结构可通过主题前缀（如 
/gov/{domain}）实现安全域的天然隔离，便于实现跨域主题的权限边界划分，
这是 Kafka 粗粒度主题、RabbitMQ 交换机 - 队列模型难以直接适配的。
3.6.2 MQTT over QUIC 传输层选型理由
　　本系统在 TLS 1.3 之外，额外选择 MQTT over QUIC 作为传输层扩展，核
心理由是解决跨域弱网下的 TCP 队头阻塞问题：
1.	无队头阻塞的多路复用：TCP 的多路复用基于字节流，单个数据包丢包会
阻塞整个连接上的所有数据传输；而 QUIC 基于流 ID 实现多路复用，单
个流的丢包不影响其他流的传输，在跨域高丢包环境中可显著提升消息交付
效率；
2.	0-RTT/1-RTT 快速握手：TCP+TLS 1.3 需要 3-RTT 完成握手，而 QUIC 仅
需 1-RTT 即可完成完整握手，0-RTT 可实现会话复用，大幅降低跨域长链
路的连接建立延迟，适配跨域节点频繁断连重连的场景；
3.	连接迁移能力：QUIC 支持连接迁移，客户端网络切换时无需重建连接，可
适配跨域异构网络的接入需求，避免因网络切换导致的消息丢失。
3.6.3 Spring Boot 3 后端框架选型理由
　　本系统选择 Spring Boot 3 作为后端管理控制平面的开发框架，核心理由如下：
1.	原生 AOT 支持与高性能：Spring Boot 3 原生支持 GraalVM 原生镜像（AOT 
编译），可将服务编译为本地可执行文件，显著降低服务的启动耗时与内存
占用，适配轻量化部署的需求；同时基于 Java 17 的新特性，服务的运行性
能相比旧版本有明显提升；
2.	成熟生态与快速开发：Spring Boot 3 具备完善的生态体系，可无缝集成 
Spring Security（身份认证）、MyBatis Plus（数据持久化）、EMQX Java SDK
（Broker 交互）等组件，大幅降低开发成本，提升迭代效率；
3.	与传统生态兼容：Spring Boot 3 保持了与 Spring 生态的兼容性，便于后续
扩展隐私计算、联邦学习等新功能，同时符合企业级应用的开发规范，便于
系统的工程落地与维护。
3.6.4 React+ECharts 前端技术栈选型理由
　　本系统选择 React 框架结合 ECharts 可视化库作为前端技术栈，核心理由
如下：
1.	组件化与响应式适配管控需求：React 的组件化特性可快速构建高交互性的
管控界面，支持主题配置、权限管理等复杂交互逻辑；响应式设计可适配不
同终端的访问需求，便于运维人员随时随地查看系统状态；
2.	ECharts 可视化适配跨域监控需求：ECharts 是国内应用最广泛的开源数据
可视化库，提供完善的拓扑图、折线图、热力图等组件，可灵活实现跨域数
据流向、消息流量、节点状态的实时可视化展示，大幅提升系统的可观测性
与可演示性，这是论文答辩与实际运维的核心需求；
3.	生态完善与组件丰富：React 生态完善，Ant Design 等组件库可提供美观、
易用的 UI 组件，降低前端开发成本，提升界面的交互性与美观度。
3.6.5 Docker 容器化技术选型理由
       本系统选择 Docker 容器技术作为基础设施层的核心，核心理由如下：
1.	跨环境一致部署适配跨域多场景：Docker 容器可将系统的所有依赖打包进
镜像，实现跨开发、测试、生产环境的一致部署，避免了 "在我机器上能跑" 
的环境问题，便于系统在不同跨域场景中快速落地；
2.	虚拟网络隔离适配安全域模拟：Docker 的网络命名空间可构建相互隔离的
虚拟网络，不同安全域的容器默认无法直接通信，仅能通过 Broker 节点实
现受控的消息交换，完美模拟真实场景中的跨域网络隔离环境；
3.	一键编排与横向扩展适配运维需求：Docker Compose 可实现多容器的一键
部署与启停，大幅降低测试环境的搭建成本；同时 Docker Swarm 或 
Kubernetes 可支持容器的横向扩展，便于系统随业务规模动态提升能力。
3.6.6 TLS 1.3+JWT 安全技术选型理由
       本系统选择 TLS 1.3 结合 JWT 令牌作为核心安全技术，核心理由如下：
1.	TLS 1.3 适配跨域端到端加密需求：TLS 1.3 相比 TLS 1.2，握手延迟更低、
加密算法更安全，可实现客户端与 Broker 之间、前后端服务之间的端到端
传输加密，确保跨域传输的数据在网络链路中不会被窃听、篡改，有效防范
中间人攻击；
2.	JWT 适配跨域无状态身份认证需求：JWT 是无状态的身份认证令牌，可将
用户身份、权限信息编码进令牌，系统无需在服务端存储会话状态，便于集
群化部署与横向扩展；同时 JWT 可结合 ACL 规则，实现跨域客户端的动
态授权与权限校验，解决不同安全域之间的身份互认问题，确保仅合法授权
的客户端可发布或订阅对应主题的消息。

4.1 跨域主题管理与路由设计
　　在跨域数据交换系统中，消息如何在相互隔离的域之间精准、安全地流转是
核心挑战之一。本节基于 MQTT 协议的层级化主题特性，设计了一套扁平、清
晰的跨域主题命名规范，并明确了核心 Broker 的路由转发机制。整体设计在满
足安全域隔离要求的前提下，尽量保持系统的轻量化与易实现性。
4.1.1 跨域主题命名规范

4.2 安全控制机制设计
　　在跨域数据交换环境中，系统面临着比单域内更复杂的安全威胁，如非法节
点接入、越权数据订阅以及中间人攻击等。为了在保障安全隔离的同时不引入过
高的耦合度，本系统设计了一套"管理面与数据面分离"的安全控制机制：利用后
端的 Spring Boot 服务负责跨域身份认证与令牌签发（管理面），利用核心 
MQTT Broker 负责高并发的连接鉴权与细粒度访问控制校验（数据面）。
4.2.1 跨域身份认证与授权机制
　　传统的基于用户名/密码的认证在跨域多节点高并发场景下，会导致中心数据
库压力过大。因此，本系统引入了基于 JSON Web Token (JWT) 的无状态认证
机制。
　　客户端在跨域接入前，必须向所属域或系统的授权中心（Spring Boot 后端）
申请接入凭证。授权中心校验其合法性后，签发包含客户端身份信息、所属安全
域及过期时间的 JWT 令牌。客户端随后以该令牌作为密码连接 MQTT Broker，
Broker 无需查询数据库，仅通过验证 JWT 的加密签名即可确认客户端身份的
合法性，大幅提升了跨域连接的认证效率。

4.3 可靠性与弱网优化设计
　　跨域数据交换环境通常涉及不同物理位置和网络域的节点，其链路往往需要
穿透多层防火墙或暴露在不稳定的公网环境中，不可避免地会面临高延迟、高丢
包、节点频繁断连等弱网问题 。为了打破传统点对点传输在此类场景下的脆弱
性，本系统基于 MQTT 协议原生的 QoS 服务质量分级机制，结合底层消息持久
化与智能重传策略，设计了一套高鲁棒性的弱网传输保障方案 。

DiskStorage --> DeadLetter : 达到最大生存时间 (TTL过期)
WaitAck --> DeadLetter : 超过最大重试次数
