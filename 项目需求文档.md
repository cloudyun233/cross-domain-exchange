# 基于发布订阅机制的数据跨域交换系统 - 项目需求文档

## 1. 项目概述

### 1.1 项目背景

本项目实现一个基于发布-订阅（Publish/Subscribe）机制的数据跨域交换平台，解决不同安全域之间的数据可控流通难题。系统在保障各安全域网络隔离边界、数据安全与访问权限可控的前提下，打破传统点对点模式的紧耦合瓶颈，实现"一处发布、多处订阅"的灵活跨域数据共享。

### 1.2 核心特性

- **异步解耦**：基于MQTT协议的发布-订阅模式，生产者与消费者无需相互知晓
- **安全隔离**：TLS加密传输 + ACL细粒度访问控制 + JWT身份认证
- **可靠传输**：三级QoS服务质量保障 + 消息持久化 + 智能重试机制
- **弱网适配**：默认使用MQTT over QUIC，应对高延迟、高丢包环境，不可用时自动回退（给出回退提示）
- **可视化监控**：实时数据流向拓扑、消息流量统计、节点状态监控

### 1.3 应用场景

- 政务协同：跨部门数据共享（如人口、社保、医疗数据交换）
- 供应链管理：上下游企业订单、库存信息实时同步
- 医疗信息共享：跨机构病历、检验数据安全流转

### 2.2 核心组件说明

当前原型默认使用 H2（MySQL 模式）进行本地演示，表结构保持与 MySQL 兼容，后续可平滑迁移到 MySQL。

## 数据库结构 (基于 H2 MySQL 模式)

- **`sys_domain`** **(安全域表)**：记录安全域ID、域编码（用于主题前缀拼接）及状态 。
- **`sys_user`** **(用户表)**：记录客户端凭证、所属域 ID 及角色类型（producer/consumer/admin） 。
- **`sys_topic_acl`** **(访问控制规则表)**：动态存储细粒度权限规则，包含主题过滤器与授权动作 。
- **`sys_audit_log`** **(审计日志表)**：流水记录连接、跨域数据交换与越权拦截日志 。

## 系统接口与类设计约束

系统代码必须遵循分层依赖与依赖倒置原则 ：

<br />

- **核心服务拆分**：鉴权与用户级 ACL 校验归属 `AuthServiceImpl`；发布代理归属 `TopicServiceImpl`；订阅与 SSE 推送归属 `SubscribeServiceImpl`；统一的 Broker 代理连接归属 `MqttClientService`；EMQX 控制面交互归属 `EmqxApiClient` 。
- **前端交互方式**：控制台通过 RESTful API 发起登录、配置、发布和管理操作，通过 SSE 接收订阅消息；前端不直接连接 MQTT Broker 。
- **扩展性接口**：预留 `DataConverter` 接口（含 `XmlDataConverter`, `JsonDataConverter` 等实现类）以及 `AuthPlugin` 接口（含 `JwtAuthPlugin`, `X509CertAuthPlugin` 实现类）

| 层级    | 组件          | 技术选型               | 职责         |
| ----- | ----------- | ------------------ | ---------- |
| 表现层   | 管理控制台       | React 18 + ECharts | 可视化监控、配置管理、SSE消息展示 |
| 业务逻辑层 | 后端服务        | Spring Boot 3.x    | 登录态管理、ACL校验、消息代理与管控 |
| 消息引擎层 | MQTT Broker | EMQX 5.0+          | 消息路由、QoS保障与Broker级能力  |
| 数据存储  | 数据库         | H2/MySQL           | 元数据与审计日志存储 |
| 基础设施  | 容器平台        | Docker             | 环境隔离与部署    |

## 3. 功能需求

### 3.1 连接管理功能

#### 3.1.1 身份认证与权限校验

- 当前默认采用用户名/密码登录，后端签发 JWT 作为前端访问后端 API 的登录态
- 发布/订阅请求在进入 Broker 前，由后端依据当前登录用户身份执行 ACL 校验
- 后端以受信任服务身份连接 EMQX，Broker 负责消息路由、QoS 和平台级 ACL 边界
- 按安全域划分身份体系，并预留 X.509 证书认证扩展接口

#### 3.1.2 连接心跳与状态维持

- 基于MQTT PINGREQ/PINGRESP心跳机制
- 连接超时自动清理
- 持久会话支持离线消息保留
- 遗嘱消息机制实现故障感知

### 3.2 消息交换功能

#### 3.2.1 主题管理与发布

- 层级化主题命名规范: `/{根节点}/{安全域标识}/{子域}/{业务模块}/{数据}`
- 通过 `/api/topics/publish` 发起发布，由后端完成 ACL 校验、格式转换后代理写入 Broker
- 支持QoS等级设置 (0/1/2)
- 支持 JSON / XML 等输入格式按需转换为 JSON 再入站

#### 3.2.2 主题订阅与消息分发

- 精确主题匹配
- 单层通配符 `+` 匹配
- 多层通配符 `#` 匹配
- 通过 `GET /api/subscribe/stream` 建立 SSE 实时通道，由后端代理订阅 EMQX
- 通过 `POST /api/subscribe/cancel` 取消订阅

#### 3.2.3 消息生命周期管理

- 当前原型的 QoS 重投递、持久会话和 Broker 持久化能力主要依赖 EMQX
- 后端负责订阅关系映射、SSE 推送和操作审计记录
- 当前版本未单独提供保留消息、消息过期和死信队列的管理界面

### 3.3 传输保障功能

| QoS等级 | 名称   | 机制                 | 适用场景              |
| ----- | ---- | ------------------ | ----------------- |
| QoS 0 | 至多一次 | 无确认，不重传            | 高频状态数据（心跳、监控采样）   |
| QoS 1 | 至少一次 | PUBLISH-PUBACK两次握手 | 通用业务数据（人口数据、订单状态） |
| QoS 2 | 恰好一次 | 四次握手去重             | 敏感关键数据（医疗病历、交易指令） |

### 3.4 系统监控与审计功能

#### 3.4.1 实时可视化监控

- 消息交换频率（TPS）
- 消息传输平均延迟
- 在线客户端数量与分域分布
- 主题数量与订阅关系拓扑
- 系统资源占用率

#### 3.4.2 全链路操作审计

- 客户端连接/断开记录
- 身份认证结果记录
- 发布/订阅请求记录
- 权限校验结果记录
- 异常行为记录

## 4. 非功能需求

### 4.1 安全性需求

1. **登录态安全**：前端用户通过用户名/密码登录，后端签发 JWT 并校验后续 API/SSE 请求身份
2. **细粒度访问控制**：主题级 ACL 支持按客户端、域、操作类型、主题范围配置；发布/订阅请求先由后端校验，再同步到 Broker
3. **未授权操作拦截**：对无权限的发布/订阅请求直接拒绝，并写入审计日志
4. **链路安全与审计**：后端到 Broker 优先使用 TLS 连接，系统保留全链路操作审计能力

### 4.2 鲁棒性与可靠性需求

1. **弱网环境可靠传输**：依托 EMQX 的 QoS 机制与后端重连逻辑，保证常见跨域弱网场景下的消息可达性
2. **单节点原型可运行**：当前项目以 Docker 单节点部署为主，不以集群高可用作为原型验收前提
3. **运行态可观测**：支持连接状态、消息流量、在线客户端和弱网模拟参数的可视化观察

### 4.3 可扩展性需求

1. **容器化部署**：Docker 容器化，Docker Compose 一键编排
2. **功能模块化**：认证插件、数据转换器、EMQX API 客户端与 MQTT 代理连接解耦
3. **前端实时通道可扩展**：当前采用 SSE，后续可扩展为 WebSocket 或前端直连 Broker 方案

## 6. 接口设计

### 6.1 RESTful API 规范

#### 6.1.1 认证接口

```
POST /api/auth/login
请求: { "username": "string", "password": "string" }
响应: { "token": "JWT令牌", "expires": 86400, "clientId": "string", "roleType": "string", "domainCode": "string", "domainName": "string" }

POST /api/auth/refresh
请求: Header: Authorization: Bearer {token}
响应: { "token": "新JWT令牌", "expires": 86400, "clientId": "string", "roleType": "string", "domainCode": "string" }
```

说明：上述 JWT 用于前端访问后端 API 以及建立 SSE 订阅流，不由浏览器直接携带去连接 Broker。

#### 6.1.2 安全域管理接口

```
GET    /api/domains              # 获取安全域列表
POST   /api/domains              # 创建安全域
GET    /api/domains/{id}         # 获取安全域详情
PUT    /api/domains/{id}         # 更新安全域
DELETE /api/domains/{id}         # 删除安全域
```

#### 6.1.3 客户端管理接口

```
GET    /api/clients              # 获取客户端列表
POST   /api/clients              # 创建客户端
GET    /api/clients/{id}         # 获取客户端详情
PUT    /api/clients/{id}         # 更新客户端
DELETE /api/clients/{id}         # 删除客户端
```

#### 6.1.4 ACL规则管理接口

```
GET    /api/acl-rules            # 获取ACL规则列表
```

# 简单演示流程：

我们将流程分为 **A端（生产者）**、**管理端（监控者）**、**B端（消费者）** 三个视角。

#### 1. 初始化准备

- **启动环境**：用户使用windows，一键启动docker，并给出登陆地址
- **准备工具**：打开4个浏览器窗口，模拟两个互不干扰的登录状态。

***

#### 2. 第一阶段：发布者（A端）—— 数据外发

> **场景**：西南医院（安全域A）需要发布一条脱敏后的病历更新。

1. **登录**：用户在 A 窗口输入账号 `producer_swu`，密码 `123456`。
2. **获取 Token**：系统调用 Spring Boot 登录接口，返回该用户的 **JWT Token**，用于后续访问后端 API 与 SSE 订阅流。
3. **进入“数据发布”页面**：
   - **选择主题**：在下拉框或树形菜单选择 `/cross_domain/medical/hosp_swu/patient/update`（注意：由于 ACL 限制，该账号只能看到其所属安全域的主题）。
   - **填写内容**：在 JSON 编辑器中输入：`{"id": "001", "action": "update", "data": "Patient_A_Record"}`。
   - **选择服务质量**：勾选 **QoS 1**（保证至少送达一次）。可下拉选择其他等级
4. **点击“立即发布”**：
   - **预期结果**：页面右上方弹出气泡提示“消息发布成功”；后端先以 `producer_swu` 身份完成 ACL 校验，再代理写入 Broker。

***

#### 3. 第二阶段：管理端（后台）—— 实时监控

> **场景**：管理员通过控制台观察数据在域间的流转情况。

1. **切换到监控大盘**：
   - 观察 **“实时拓扑图”**：发现安全域 A 的节点图标从红色变为绿色，并有一条闪烁的虚线指向中心 Broker。还可以看到上线的发布者订阅者。上线为绿，不在线为红
   - 明确展示A端一条消息同时到达B端和C端的图形化演示线
   - 观察 **“流量统计”**：MQTT 消息入站数（Inbound）实时加 1。并显示当前后端到 Broker 的连接协议（TLS/TCP）
   - 有“tc模拟跨域弱网”的开关，管理者可以模拟延迟和丢包来测试qos。
2. **查看“审计日志”**：
   - 记录显示：`[2026-03-31 10:00:01] 用户 producer_swu 向主题 /cross_domain/medical/hosp_swu/patient/update 发布消息成功，ACL 校验通过。`

#### 4. 第三阶段：订阅者（B端）—— 跨域接收

> **场景**：社保局（安全域B）需要实时接收来自医院的更新。

1. **登录**：用户在 B 窗口登录账号 `consumer_social`。
2. **进入“数据订阅”页面**：
   - **配置订阅**：输入或选择主题 `/cross_domain/medical/#`（使用通配符，订阅医疗域下所有子项）。
   - **建立连接**：点击“开始监听”，前端建立 SSE 连接，后端代理向 Broker 发起订阅。
3. **接收数据**：
   - **预期结果**：下方“消息接收列表”中立刻跳出刚才 A 端发送的 JSON 包。
   - **自动转换验证**（针对异构需求）：如果 A 端发的是 XML，B 端此时应在列表里看到后端转换后的 JSON，并可在消息中查看 `_meta.source_format` 等来源标记。
4. 另有一个订阅者C，默认没有订阅这个主题的权限，但是可以让管理人员在管理端操控，动态增加/修改人员acl权限
